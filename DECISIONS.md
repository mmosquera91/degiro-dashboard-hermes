# DECISIONS.md — Architecture Decisions

## `network_mode: host` in Docker Compose

**Date:** 2026-05-06
**Status:** Decided

---

### Decision

Use `network_mode: host` in `docker-compose.yml` instead of port publishing (`ports: - "8000:8000"`).

### Pros

- **No NAT overhead** — direct access to host network stack; lower latency for API calls
- **Simplifies service discovery** — services reference each other as `localhost` without needing explicit `expose` declarations
- **Easier dev UX** — browser devtools connect to `localhost:8000` without additional configuration
- **Works with existing `app.listen(8000)` pattern** — no need to change the application bind address

### Cons

- **No port-level access control** — Docker cannot enforce per-port rules; all host ports are accessible
- **Harder to run multiple stacks** — cannot have two `host` mode containers compete for the same port
- **Container isolation reduced** — container has full host network stack access (though this is already the case with volume mounts)
- **Mac/Windows Docker Desktop** — `host` networking is not supported; must use `ports:` or a custom bridge network in those environments
- **CI/CD complexity** — some CI environments (e.g., CodeBuild, GitHub Actions with Docker) do not support `host` mode

### Mitigation for cons

- `network_mode: bridge` with explicit `ports:` is the fallback for non-Linux environments
- A `.env` variable `NETWORK_MODE=host|bridge` can toggle between the two modes
- Security is enforced at the application level via `BROKR_AUTH_TOKEN` bearer auth on all `/api/*` routes
